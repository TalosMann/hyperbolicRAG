# Implement Cog‑RAG and Cog‑RAG‑Flash in HyperScholar

This document provides the complete blueprint to add two new RAG backends to HyperScholar: **PureCogRAGBackend** (faithful to the Hu et al. paper) and **CogRagFlashBackend** (a cost‑effective variant). Both are drop‑in compatible with the existing `RAGBackend` ABC, storage layer, GUI, and evaluation framework.

---

## 1. Goal

Integrate both Cog‑RAG variants into the HyperScholar codebase so that:

- They are selectable in `config.yaml` and the GUI.
- They share the same storage contracts (`kv_cls`, `vector_cls`, `embedder`, `llm_func`).
- The evaluation framework can capture provenance from both (for later visualisation).
- Indexing and query work seamlessly alongside HyperRAG and HierarchicalRAG.

---

## 2. Overview of the Two Backends

| Backend | Indexing Cost | Query Cost | Fidelity to Paper |
|---------|---------------|------------|-------------------|
| **PureCogRAG** | Very high (~4 LLM calls per chunk) | Moderate (two‑stage with diffusion) | 100% |
| **CogRagFlash** | Low (1 LLM call per geometric cluster, like HierarchicalRAG) | Slightly higher (fast LLM for targeting) | Conceptual (two‑stage cascade, but themes are geometric summaries) |

**Core ideas:**

- **PureCogRAG**: dual‑hypergraph (theme + entity), all extraction done at index time.
- **CogRagFlash**: reuses `_greedy_clusters` to build thematic summaries; uses a cheap LLM at query time to generate targeting keywords, then searches raw chunks.

## 4. Implementation Details

### 4.1 PureCogRAGBackend (`rag/pure_cograg_backend.py`)

**Prompts** (from Appendix F of the paper, adapted to match your `llm_func` signature):

- `P_EXT_THEME`: summarise the primary theme of a chunk.
- `P_EXT_KEY`: extract key entities related to that theme.
- `P_EXT_ENTITY`: extract all fine‑grained entities with descriptions.
- `P_EXT_REL`: extract high‑order relations (entity sets).
- `P_KEYWORD_THEME`: from query, extract abstract theme keywords.
- `P_ALIGN_ENTITY`: from query + theme answer, extract concrete entity keywords.
- `P_THEME_ANSWER`: initial answer based on theme hyperedges.
- `P_FINAL_ANSWER`: final answer using entity details.

**Storage** (within the tenant namespace):

- `theme_edges_vdb` (vector) – stores theme hyperedge descriptions; metadata: `incident_nodes` (list of key entity names), `chunk_id`.
- `theme_nodes_kv` – key‑value store for incidence: key = hash(name), value = `{"name": ..., "incident_edge": ..., "chunk_id": ...}`.
- `entity_nodes_vdb` (vector) – stores entity descriptions; metadata: `incident_edges` (list of high‑order edge IDs), `chunk_id`.
- `entity_edges_kv` – key‑value store for high‑order relations; key = hash(description), value = `{"content": ..., "chunk_id": ...}`.

**Indexing**:
- Chunk the document (use `_chunk_text`).
- For each chunk:
  - Extract theme → store as a hyperedge in `theme_edges_vdb`.
  - Extract key entities → for each, store in `theme_nodes_kv` with the edge ID.
  - Extract entities → store each as a vector in `entity_nodes_vdb`.
  - Extract high‑order relations → store each in `entity_edges_kv`; then for each entity in that set, append the edge ID to its `incident_edges` list (merge if multiple edges).

**Query** (two‑stage):
- **Stage 1 (Theme)**:
  - Extract theme keywords from query.
  - Vector‑search `theme_edges_vdb` with those keywords (top‑k=3).
  - Diffuse: for each hit, get its `incident_nodes`, look up each node in `theme_nodes_kv` to collect names.
  - Generate `a_theme` using `P_THEME_ANSWER` with the combined context.
- **Stage 2 (Entity)**:
  - Extract aligned entity keywords using `P_ALIGN_ENTITY` with original query and `a_theme`.
  - Vector‑search `entity_nodes_vdb` (top‑k=6).
  - Diffuse: for each hit, get its `incident_edges`, look up those edge descriptions in `entity_edges_kv`.
  - Generate final answer with `P_FINAL_ANSWER` using the entity contexts and `a_theme`.

**Return**: `QueryResult` with `answer`, `sources` (from entity nodes), and `ok=True/False`.

### 4.2 CogRagFlashBackend (`rag/cograg_flash_backend.py`)

**Philosophy**: Reuse the existing hierarchical indexing pipeline (chunking + greedy clustering) but store only **level‑1 summaries** as a thematic overlay. At query time, retrieve themes, use a cheap LLM to extract targeting keywords, then search raw chunks exclusively.

**Storage**:
- `raw_chunks_kv` + `raw_chunks_vdb` – same as in HierarchicalRAG (could reuse the shared content stores to avoid duplication, but for isolation we keep separate namespaces).
- `theme_targeting_vdb` – vector store of level‑1 summaries; metadata: `summary_text`, `child_chunks` (list of chunk IDs).

**Indexing**:
- Chunk all documents → store raw chunks.
- Compute embeddings for all chunks.
- Run `_greedy_clusters` on the embeddings.
- For each cluster, generate a summary using the heavy LLM (`SUMMARY_PROMPT`).
- Store each summary in `theme_targeting_vdb` with its child chunk IDs.

**Query** (two‑stage):
- **Stage 1**:
  - Search `theme_targeting_vdb` with the query (top‑k=2) to get macro summaries.
  - Use a **fast LLM** (`llm_fast_func`) with `TARGETING_RECON_PROMPT` to produce 4‑5 concrete keywords based on the summaries and the query.
- **Stage 2**:
  - Enrich the original query with those keywords.
  - Vector‑search `raw_chunks_vdb` with the enriched query (top‑k=8).
  - Retrieve full chunk contents from `raw_chunks_kv`.
  - Feed only raw chunks (no summaries) into the heavy LLM (`ANSWER_PROMPT`) to generate the final answer.

**Return**: `QueryResult`.

### 4.3 Updates to `eval/provenance.py`

To support the evaluation framework, add two provenance‑capture functions:

- `pure_cograg_query_with_provenance(backend, query, top_k)`:
  - Calls `backend.query` but also records the theme hyperedges, diffused key entities, entity nodes, and diffused high‑order edges.
  - Normalises to `{"answer": ..., "ok": ..., "provenance": {"theme_edges": [...], "theme_nodes": [...], "entity_nodes": [...], "entity_edges": [...]}}`.

- `cograg_flash_query_with_provenance(backend, query, top_k)`:
  - Records the theme summaries used, the targeting keywords generated, and the final raw chunks retrieved.
  - Provenance shape: `{"theme_summaries": [...], "targeting_keywords": ..., "chunks_accessed": [...]}`.

Then update `eval/runner.py` to use these functions when `backend.name` matches.

## 10. Notes for the AI Assistant

- All storage classes (`kv_cls`, `vector_cls`) are asynchronous and have methods `upsert`, `query`, `get_by_id`, `all_keys`, `filter_keys`, `drop`, and an `index_done_callback` to persist to disk. Use them consistently.
- The `llm_func` is a callable that accepts a prompt string and `hashing_kv` (cache). It returns a string.
- The embedding dimension is 1024 (BGE‑M3). Use the `embedder` callable with a list of texts.
- When storing vectors, ensure the content field is always present; metadata can be arbitrary JSON‑serialisable.
- For PureCogRAG, when updating `incident_edges` for entity nodes, you may need to merge existing lists (check `get_by_id` first). Use `upsert` to overwrite.
- For provenance, the eval framework expects the `QueryResult` to have `sources` (a list of `SourceRef`) and `ok` boolean. The `answer` field is the final text.
- The `working_dir` for each backend is `hyperscholar_runtime/<backend_name>/<namespace>/` – create it if needed.
